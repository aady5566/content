# 期末專案- RPG 遊戲設計 (12/23 - 01/06)

# 簡報區塊 1：專案核心目標 (Project Goal)

## 投影片 1.1：我們要做什麼？ (What are we building?)

* **標題**：專案目標：打造 AI 驅動的歷史互動 RPG 遊戲
* **核心概念**：
    * 結合所有上課所學的提示詞工程、文獻、事實查核、Gemini 應用程式產出的互動網頁。
    * 利用 AI 工具鏈，將創意轉化為實際可玩的程式。
* **最終交付規格 (The Deliverable)**：
    * **格式**：Standalone HTML (單一 HTML 檔案)。
    * **特性**：
        * 無需安裝、無需伺服器。
        * 點開即玩 (Plug & Play)。
        * 包含完整的 CSS 樣式與 JavaScript 互動邏輯。
* **🎤 講師口語提示 (Speaker Notes)**：
    「各位同學，期末我們要做的不只是一份報告，而是一個『真的能玩』的遊戲。不用擔心程式太難，我們這次的目標是產出一個『Standalone HTML』檔案，也就是說，你做完後傳給朋友，他只要有瀏覽器點開就能直接玩，完全不用安裝。」

## 投影片 1.2：透過專案我們會學會什麼？ (Core Skills)
這頁解釋「為什麼要做這個」，強調本次專案的三大技術支柱。

* **標題**：三大核心能力養成
* **1. AI 角色賦魂 (Character Design)**
    * **技術點**：Prompt Engineering + MBTI。
    * **內容**：學習如何設定 Prompt，結合 MBTI 人格架構，讓遊戲 NPC 擁有獨特個性與反應，不再是只有固定台詞的機器人。
* **2. 深度敘事與考據 (Narrative & Research)**
    * **技術點**：NotebookLM + 歷史資料探索。
    * **內容**：利用 NotebookLM 消化大量歷史文獻，生成符合 **歷史脈絡 (Historical Context)** 的劇本，確保故事既有創意又有憑據。
* **3. 互動介面實作 (Interface Implementation)**
    * **技術點**：Gemini Canvas + 前端整合。
    * **內容**：將生成的故事與角色，透過 Gemini Canvas 協助撰寫程式碼，整合為完整的 RPG 互動介面。
* **🎤 講師口語提示 (Speaker Notes)**：
    「這個專案會訓練大家三個很值錢的能力：第一，怎麼用 Prompt 和 MBTI 讓 AI 扮演特定角色。第二，怎麼用 NotebookLM 幫你讀歷史資料，寫出有深度的劇本，而不是瞎掰。第三，最後怎麼用 Gemini Canvas 幫你寫程式，把這些變成一個漂亮的網頁介面。」

# 簡報區塊 2：遊戲設計重點 (Design Focus)

## 投影片 2.1：四大設計支柱 (The 4 Pillars)
這頁定義遊戲必須包含的功能要素。

* **標題**：遊戲設計規範：必須包含的要素
* **1. N+1 變數系統**：
    * 除了基本的 A/B 選項 (N)，後台必須運算至少一組 **隱藏數值 (+1)**（如：好感度、體力值、金錢、道德值等），並據此影響結局。
* **2. 歷史考據**：
    * 劇情內容需基於真實歷史背景（如特定朝代、事件或法條），不可完全架空。
* **3. 勝敗判定**：
    * 遊戲必須有明確的 **勝利 (Win)** 與 **失敗 (Loss/Game Over)** 條件，而非只有單一結局。
* **4. 工具應用**：
    * 使用指定的 GenAI 工具輔助生成程式碼、文本與美術素材。
* **🎤 講師口語提示 (Speaker Notes)**：
    「你們的遊戲不能只是點選項看故事。後台必須要有『數學』在運作。比如玩家選了某個選項，表面上沒事，但後台的『民心值』扣了 10 點，當民心低於 0 的時候，遊戲就會強制結束。這就是 N+1 變數系統。」

# 簡報區塊 4：遊戲規則 (Game Rules)

## 投影片 4.1：核心互動迴圈 (Core Loop)
這頁解釋遊戲運行的基本邏輯結構。

* **標題**：遊戲邏輯架構圖
* **流程步驟**：
    1.  **情境導入**：顯示歷史背景與當下事件。
    2.  **玩家決策**：提供 2-3 個選項。
    3.  **變數運算 (N+1)**：依據選擇更新後台數值 (例：資金 -50, 軍心 +10)。
    4.  **判定檢核**：檢查數值是否觸發結局 (例：資金 <= 0 ➔ 破產結局)。
    5.  **敘事推進**：進入下一個情境或結局畫面。
* **🎤 講師口語提示 (Speaker Notes)**：
    「這就是你們要寫給 AI 的核心邏輯：顯示情境 -> 玩家選擇 -> 更新數值 -> 檢查是否輸了 -> 繼續下一個情境。請確保你們的 Prompt 清楚描述這個迴圈。」

# 簡報區塊 5：範例介紹 (Examples)

## 投影片 5.1：黃金範例 (Gold Standard)
這頁提供具體參照物，讓學生理解高標在哪裡。

* **標題**：參考範例：《民國律法情緣》
* **專案概述**：
    * 背景：民國初年，新舊法交替時期。
    * 核心衝突：傳統家庭觀念 vs 現代個人法治權益。
* **技術拆解**：
    * **變數**：MBTI (T/F 維度) + 法治觀念值 (0-100)。
    * **考據**：引用真實《中華民國民法》親屬篇條文。
    * **判定**：
        * 若「法治值 < 50」且強行離婚 ➔ 判定敗訴 (社會性死亡)。
        * 若「法治值 > 80」且引用正確法條 ➔ 判定勝訴 (獨立新女性)。
* **🎤 講師口語提示 (Speaker Notes)**：
    「看看這個範例，《民國律法情緣》好玩的地方在於它不是亂選。你必須了解當時的法律（考據），並且累積足夠的『法治值』（變數），才能打贏離婚官司（勝敗判定）。這就是我們要求的深度。」

## 投影片 6.2：宿敵系統 (The Rivalry System)
這頁介紹本專案最特殊的機制——強制尋找競爭對手。

* **標題**：建立競爭關係：尋找你的宿敵

    * **三組對手原則 (Rule of Three)**：

# 簡報區塊 8：評分與驗收 (Grading & Criteria)

## 投影片 8.1：雙維度評分模型 (Scoring Model)
這頁解釋分數的構成，平衡「製作水準」與「使用者回饋」。

* **標題**：評分標準：品質與市場的平衡
* **A. 製作質量 (Production Quality) - 60%**：
    * **技術實踐**：變數邏輯是否順暢？有無 Bug？
    * **敘事深度**：歷史考據是否紮實？劇本是否合理？
    * **美術/UI**：視覺風格與使用者體驗。
* **B. 競技結果 (Competitive Result) - 40%**：
    * **來源**：來自你鎖定的 **「3 組對手」**。
    * **依據**：得分矩陣 (The Scoring Matrix)。
* **🎤 講師口語提示 (Speaker Notes)**：
    「分數由兩部分組成：60% 看製作水準，程式穩不穩、歷史有沒有亂寫。剩下的 40% 取決於對手的評分。如果遊戲體驗不佳，或 Bug 太多導致無法遊玩，這 40% 的分數將會受到影響。」

## 投影片 8.2：得分矩陣 (The Scoring Matrix)
這頁具體說明對手互評時的標準。

* **標題**：對手互評視角：評分的三大維度
* **互評指標**：
    1.  **好玩 (Fun Factor)**：
        * 遊戲過程是否有趣？是否具備重玩價值？
    2.  **驚艷 (Creative Factor)**：
        * 變數設計是否有創意？是否展現了技術亮點？
    3.  **完整 (Completion Factor)**：
        * 流程是否完整？有沒有嚴重的邏輯漏洞？
* **🎤 講師口語提示 (Speaker Notes)**：
    「互評時，請依據這三個標準給分。為了拿到高分，建議大家專注在『變數設計』——讓玩家的選擇真正影響結局，這是最能展現創意的部分。」
